📝 總體評估

📋 Executive Summary

專案概況：GenAI_DrawToPlay 是一個小型 AI 繪圖到 Spotify 推薦 app：前端是單頁 vanilla JS/canvas UI，後端是 FastAPI + LangChain/OpenAI + Spotify + PostgreSQL，部署線索包含 Dockerfile、AWS App Runner yaml 與 ECR push script。

主要風險：目前最大的 release blocker 是 frontend/app.js 使用未宣告的 API_BASE，核心 chat/song/session API flow 會在瀏覽器執行期失敗。第二層風險是匿名公開端點仍能觸發 LLM/Spotify 成本路徑，且 rate limiter 直接信任 X-Forwarded-For。

亮點：repo 小、邊界大致可讀，SQL 使用 parameterized query，Spotify HTTP call 有 timeout，最近已加入 CORS env、payload 長度限制、session TTL、DOM 安全渲染與 DB context manager 等改善。

優先修復：先修 API_BASE 與 production ALLOWED_ORIGINS，再補 signed session/bot challenge、blocking I/O 處理、critical path tests/CI、DB migrations 與 privacy/accessibility 基線。

總評：整體 4.9 / B。這不是不可救的系統，但目前還不適合穩定 public release；修復重點很集中，先補 release blocker 與測試後健康度會快速上升。

🗺️ Project Map

主要元件：frontend/index.html/style.css/app.js 提供 canvas 繪圖、chat dialog、Spotify card 與 rating；backend/main.py 提供 FastAPI endpoints；backend/agent.py 管理 prompt、session memory 與 LLM JSON 產生；backend/tools.py 封裝 Spotify；backend/database.py 寫入 PostgreSQL sessions。

資料流：使用者選色與畫圖 → app.js capture canvas + telemetry → /chat 呼叫 GPT-4o cold read → 使用者回覆 → /song-result 透過 LLM 產生 search query，再查 Spotify → 前端顯示 card → /session-complete 儲存 strokes、telemetry、conversation、推薦結果 → /rate 與 /spotify-opened 補寫互動資料。

狀態所有權：前端 app.js 持有 workflow state、strokes、telemetry、dbSessionId；後端 agent.py 持有 in-memory conversation；main.py 持有 rate-limit buckets 與 frontend session 到 db id 的 process-local mapping；PostgreSQL 是 session analytics 的長期儲存。

外部介面：OpenAI/ChatOpenAI、Spotify Accounts/Search/Artists API、ip-api geolocation、PostgreSQL、AWS App Runner/ECR、Google Fonts。

🔍 Detailed Findings

1. 前端使用未定義的 API_BASE，所有後端互動在執行期失敗

CriticalConfirmed
📁 Stability / Release📍 frontend/app.js:1-3, 955, 1000, 1077, 1114, 1139, 1161, 1201🎯 fetch calls / API bootstrap
問題：目前頁面載入後不會立刻語法失敗，但使用者一旦完成繪圖、送 chat、揭露歌曲、儲存 session 或評分，瀏覽器會拋出 ReferenceError: API_BASE is not defined。這會讓核心產品流程完全無法呼叫後端。
為什麼重要：這是 public release 的阻斷缺陷：AI 對話、Spotify 推薦、session 儲存與 rating 都依賴這個變數。
真實失敗情境：使用者畫完圖按 Completed，openChat() 進入 try block 後執行 fetch(`${API_BASE}/chat`)，JS 在組字串時丟 ReferenceError，畫面只顯示錯誤訊息或卡住，沒有推薦結果。
File: frontend/app.js:1-3, 955, 1000, 1077, 1114, 1139, 1161, 1201 Function / Module: fetch calls / API bootstrap Relevant behavior: 檔案只宣告 DEPLOYED_API_BASE 與 SESSION_ID，後續所有 fetch 都使用未宣告的 API_BASE。
最小修復 / 長期修復
Minimal fix: 定義 API_BASE，例如：const API_BASE = /^localhost$|^127\.0\.0\.1$/.test(location.hostname) ? 'http://localhost:8080' : DEPLOYED_API_BASE; Better long-term fix: 將 API base 移到 build/deploy config，並在 CI 中加入 smoke test 檢查 window.API 或 fetch config 已定義。
⏱ 預估： 15-30 分鐘  ·  回歸測試： 用 Playwright 載入 frontend/index.html，mock backend，完成繪圖並點 Completed，斷言 /chat request 被送出且 console 沒有 ReferenceError。

2. 匿名 LLM/Spotify 成本路徑仍可被繞過 rate limit 濫用

HighConfirmed
📁 Security / Cost / AI-Safety📍 backend/main.py:61-73, 119-140; backend/agent.py:410-411, 528-555🎯 rate_limit, /chat, /song-result, score_song_candidates
問題：雖然已有 per-IP rate limit，但若服務直接接受外部請求或 proxy 未清洗 X-Forwarded-For，攻擊者可以變換 header 繞過限制，持續觸發 OpenAI 與 Spotify API。CORS 不能阻止 server-to-server 或 curl 類請求。
為什麼重要：這會造成帳單暴增、API quota 被耗盡，也會讓正常使用者拿不到推薦。
真實失敗情境：攻擊者寫腳本每次帶不同 X-Forwarded-For 呼叫 /chat 和 /song-result；每次請求都穿過 rate_limit bucket，進入 llm.invoke 與 Spotify search。
File: backend/main.py:61-73, 119-140; backend/agent.py:410-411, 528-555 Function / Module: rate_limit, /chat, /song-result, score_song_candidates Relevant behavior: 所有昂貴端點仍是匿名公開；rate limiter 以 request header x-forwarded-for 作為 key，且 /chat、/song-result 會觸發 LLM 呼叫。
最小修復 / 長期修復
Minimal fix: 只在可信 proxy 後才讀 X-Forwarded-For；否則使用 request.client.host。對 /chat、/song-result、/playlist-result 加 server-issued signed session token 或 Turnstile 驗證。 Better long-term fix: 在 API Gateway/WAF 或 Cloudflare 層加 global quota、bot challenge、per-session budget 與 OpenAI spend alert。
⏱ 預估： 0.5-1 天  ·  回歸測試： 新增 integration test：同一 client IP 但偽造多個 X-Forwarded-For 時仍被同一 rate bucket 限制；缺少 signed session token 時 /chat 回 403。

3. FastAPI async handlers 內執行同步 LLM、HTTP 與 DB I/O，會阻塞 event loop

HighConfirmed
📁 Stability / Performance📍 backend/main.py:119-140, 208-266; backend/agent.py:410-411, 447-555; backend/tools.py:12-17, 34-38, 65-69, 98-102, 114-118; backend/database.py:9-16🎯 FastAPI endpoints, agent, tools, database
問題：在 async web server 中，這些同步操作會佔住 event loop worker。當 OpenAI、Spotify、ip-api 或 DB 慢時，其他使用者請求也會排隊，造成 tail latency 或 timeout。
為什麼重要：這是 public app 的可靠性與 scalability 風險，流量稍高或外部 API 變慢時會放大成整體服務延遲。
真實失敗情境：Spotify token endpoint 延遲 10 秒時，一個 /song-result request 阻塞 event loop；同 worker 上的 /health 或 /chat 也被延遲。
File: backend/main.py:119-140, 208-266; backend/agent.py:410-411, 447-555; backend/tools.py:12-17, 34-38, 65-69, 98-102, 114-118; backend/database.py:9-16 Function / Module: FastAPI endpoints, agent, tools, database Relevant behavior: async def endpoint 直接呼叫同步 llm.invoke、requests.get/post 與 psycopg2.connect/query。
最小修復 / 長期修復
Minimal fix: 把 endpoint 改成同步 def 讓 FastAPI 放到 threadpool，或用 anyio.to_thread.run_sync 包住 blocking sections。 Better long-term fix: 改用 async OpenAI/client、httpx.AsyncClient、asyncpg/psycopg pool，並配置 connection pool、timeouts、bulkhead。
⏱ 預估： 0.5-2 天  ·  回歸測試： 用測試 stub 將 Spotify call sleep 2 秒，同時發兩個 request，斷言第二個 request 不被第一個同步 call 阻塞。

5. process-local session 與 db id 對應在重啟或多 instance 時失效

MediumConfirmed
📁 Stability / Data-Integrity📍 backend/main.py:27-51, 245, 249-266; backend/agent.py:13-16, 341-365🎯 _session_db_ids, session_memories
問題：App Runner/ECS 可能重啟、scale out 或開多 worker；使用者若在 session save 後被導到另一個 instance，/rate 或 /spotify-opened 會因本機 _session_db_ids 沒資料而 403。chat history 也會在重啟或跨 instance 時消失。
為什麼重要：這會造成 analytics 資料不完整、評分遺失，以及使用者在正常部署事件後遇到隨機失敗。
真實失敗情境：session-complete 在 instance A 回傳 db id；使用者 2 秒後點星星，load balancer 把 /rate 導到 instance B，B 沒有該 session mapping，回 403。
File: backend/main.py:27-51, 245, 249-266; backend/agent.py:13-16, 341-365 Function / Module: _session_db_ids, session_memories Relevant behavior: session memory 與 db_session_id link 都保存在單一 Python process 記憶體。
最小修復 / 長期修復
Minimal fix: 把 session_id 到 db_session_id 的校驗改為查 DB：WHERE id=%s AND frontend_session_id=%s。 Better long-term fix: 把 conversation/session state 移到 Redis 或 DB，設定 TTL，並支援多 instance。
⏱ 預估： 0.5-1 天  ·  回歸測試： integration test：先寫入 session，再清空 _session_db_ids 或模擬新 app process，/rate 仍能用 DB 關聯驗證成功。

7. 模型 JSON fallback 會接受夾帶文字中的外層物件，輸出驗證過寬

MediumConfirmed
📁 AI-Safety / Type-Safety📍 backend/agent.py:323-338, 425-563🎯 _parse_json_response and model result functions
問題：這讓模型可以在非 JSON 文字裡夾帶任意 JSON 物件而被接受，且沒有 schema 驗證 scores 長度、欄位型別、URL/語言等 contract。
為什麼重要：AI 輸出本身是低信任邊界；寬鬆解析會讓 prompt drift 或 injection 更難被發現，並把錯誤資料送到 Spotify/DB。
真實失敗情境：模型回覆「Sure, here is why... {"queries":[...]}」或含有兩個 JSON 片段，fallback 可能接受非預期外層物件，playlist/song result contract 仍被當成功。
File: backend/agent.py:323-338, 425-563 Function / Module: _parse_json_response and model result functions Relevant behavior: 若 json.loads 失敗，_parse_json_response 會抓第一個 { 到最後一個 } 再解析。
最小修復 / 長期修復
Minimal fix: 移除 outermost brace fallback，使用 Pydantic model 驗證每個 LLM JSON response，缺欄位或多餘文字直接回 structured error。 Better long-term fix: 使用 OpenAI structured outputs / JSON schema，並加 eval cases 覆蓋 malformed output、prompt injection 和語言切換。
⏱ 預估： 0.5-1 天  ·  回歸測試： unit test：餵入含前後文字、缺欄位、錯型別、scores 長度錯誤的 response，斷言解析失敗且不呼叫 Spotify。

8. 使用者心理/行為資料與 IP geolocation 缺少保留、刪除與同意邊界

MediumConfirmed
